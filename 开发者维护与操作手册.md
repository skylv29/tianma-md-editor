## 壹 | 读懂项目：AI 辅助接手指南

> 如果你（或 AI）隔了很长时间没碰这个项目，以下是最高效的"重新上手"路径。  
> 按优先级顺序把这些文件内容提供给 AI，基本可以还原全部编程信息。

### 文件阅读优先级排序

| 优先级 | 文件路径 | 读它能了解什么 |
|:---:|---|---|
| ⭐⭐⭐ | `package.json` | 项目名称、版本、全部依赖、npm 脚本、Electron Builder 打包配置，是项目的"说明书封面" |
| ⭐⭐⭐ | `src/App.jsx` | 全部业务逻辑：三种视图模式、所见即所得装饰引擎、公文导出到 Word、键盘快捷键、IPC 通信调用 |
| ⭐⭐⭐ | `main.cjs` | Electron 主进程：窗口创建、文件打开/保存/另存为、脏状态标题栏、关闭前保存询问、双击打开文件 |
| ⭐⭐ | `src/index.css` | 所有 UI 样式：顶栏/侧边大纲/编辑区/预览区/底部状态栏/模态框，以及 `.prose` 公文排版规则 |
| ⭐⭐ | `vite.config.js` | 构建配置，最关键的是 `base: './'`（解决 Electron 打包后白屏问题） |
| ⭐ | `preload.cjs` | 渲染进程与主进程之间的桥梁（contextBridge 暴露 `window.electronAPI`）。文件读写出问题时先看这里 |

## 贰 | 新环境部署与运行指南

> 换了新电脑，或刚从 GitHub 克隆项目时，由于 `node_modules` 不在仓库里，需要重新配置环境。  
> **请从第一步开始，严格按顺序执行，不要跳过任何步骤。**

### 第一步：安装基础软件

确保以下三个软件已安装，缺一不可：

| 软件 | 说明 | 下载地址 |
|---|---|---|
| **Node.js** | 推荐 v18 LTS 或更高版本 | https://nodejs.org |
| **Git** | 版本控制工具 | https://git-scm.com |
| **VS Code** | 推荐的开发编辑器 | https://code.visualstudio.com |

安装完成后，**必须重启 VS Code**，否则终端可能无法识别新安装的命令。

验证安装是否成功（在终端输入以下命令，有版本号输出即为正常）：

```powershell
node -v
git --version
```

### 第二步：配置国内镜像源（必须最先做，不做必卡）

打开 VS Code，按 `Ctrl + ~` 打开终端，**确认终端类型为 PowerShell**（右上角下拉可切换）。

**依次运行以下两条命令：**

```powershell
# 第 1 条：切换 npm 为国内源，加速所有插件下载
npm config set registry https://registry.npmmirror.com

# 第 2 条：指定 Electron 国内镜像（最常见的卡死原因，必须设置）
$env:ELECTRON_MIRROR="https://npmmirror.com/mirrors/electron/"
```

验证第 1 条设置成功：

```powershell
npm config get registry
# 输出应为：https://registry.npmmirror.com/
```

> ⚠️ **注意**：`$env:ELECTRON_MIRROR` 只对当前终端会话有效，每次重新打开 VS Code 后，凡是要执行 `npm install` 或打包，都需要重新运行第 2 条命令。第 1 条设置是永久生效的，只需设置一次。

### 第三步：拉取项目代码

如果还没有把代码下载到本地，在终端中执行（替换为你的实际仓库地址）：

```powershell
git clone https://github.com/yourname/tianma-md-editor.git
cd tianma-md-editor
```

如果已有代码压缩包，解压后在 VS Code 中打开该文件夹，再打开终端继续操作。

### 第四步：安装依赖

```powershell
npm install
```

这一步会下载所有依赖到 `node_modules/` 文件夹，正常耗时 2~5 分钟。终端最后显示 `added xxx packages` 即为成功，过程中出现黄色 `warn` 警告是正常现象，不影响使用。

### 第五步：启动开发模式

```powershell
npm run electron:dev
```

终端会先启动 Vite 编译服务，几秒后自动弹出 Electron 窗口。此模式下修改 `.jsx` 或 `.css` 文件后保存，窗口内容会自动热更新。

### 第六步：打包为安装程序（需要时执行）

如果需要生成 `.exe` 安装包：

```powershell
# 每次新开终端都要重新设置 Electron 镜像
$env:ELECTRON_MIRROR="https://npmmirror.com/mirrors/electron/"

npm run electron:build
```

打包完成后，在项目根目录的 `dist_electron/` 文件夹下可以找到 `天马md编辑器 Setup x.x.x.exe`。

### 遇到白屏时的调试方法

在 `main.cjs` 中找到以下代码，去掉注释符号 `//`，重新打包后软件启动时会自动打开开发者工具，可以直接看到具体报错：

```js
// mainWindow.webContents.openDevTools();
// ↓ 改为：
mainWindow.webContents.openDevTools();
```

排查完毕后记得重新注释回去再发布。

## 叁 | 项目瘦身、清理与打包指南

> 长期开发或多次打包失败后，项目目录会积累大量临时文件，总大小可能超过 1GB。  
> 需要把源码发给别人或硬盘空间不足时，执行以下清理操作。

### 认识"重资产"目录

| 目录 | 说明 | 体积（约） | 能否恢复 |
|---|---|---|---|
| `node_modules/` | 全部 npm 依赖包 | 500MB ~ 1GB | ✅ `npm install` 即可恢复 |
| `dist_electron/` | electron-builder 打包产物 | 50 ~ 200MB | ✅ 重新打包恢复 |
| `dist/` | Vite 编译的前端静态文件 | 5 ~ 30MB | ✅ `npm run build` 恢复 |

### 可能存在的冗余文件

| 文件 | 说明 |
|---|---|
| `main.js` | 若已有 `main.cjs` 且运行正常，此旧文件可删除 |
| `builder-debug.yml` | 打包失败时留下的调试日志 |
| `builder-effective-config.yaml` | 打包过程中生成的临时配置文件 |

### 一键强力清理（PowerShell）

```powershell
# 第 0 步：强制结束可能占用文件的残留进程（遇到"文件被占用/访问被拒绝"时先运行这步）
taskkill /F /IM "天马md编辑器.exe" /T; taskkill /F /IM electron.exe /T

# 第 1 步：删除所有可再生的大体积目录
Remove-Item -Recurse -Force .\node_modules, .\dist_electron, .\dist -ErrorAction SilentlyContinue

# 第 2 步：清理冗余文件
Remove-Item -Force .\main.js, .\builder-debug.yml, .\builder-effective-config.yaml -ErrorAction SilentlyContinue
```

清理完成后，项目源码通常只有几百 KB，可以直接压缩发送或归档备份。

### 千万不要删除的文件

```
src/              ← 全部源代码
public/           ← 图标等静态资源
package.json      ← 项目配置与依赖声明
package-lock.json ← 依赖版本锁定文件
main.cjs          ← Electron 主进程
preload.cjs       ← IPC 桥接脚本
index.html        ← 应用入口
vite.config.js    ← 构建配置
```

### 清理后重新打包的完整流程

```powershell
# 1. 设置 Electron 镜像（新终端必须重新设置）
$env:ELECTRON_MIRROR="https://npmmirror.com/mirrors/electron/"

# 2. 重新安装依赖
npm install

# 3. 打包
npm run electron:build
```

## 肆 | 如何更新版本号及软件名

所有版本和名称信息集中在 **`package.json`** 中，通常只需改这一个文件。

### 修改版本号

```json
{
  "version": "1.1.0"
}
```

版本号建议遵循语义化规范（SemVer）：

- `1.0.x`：修复 Bug（Patch）
- `1.x.0`：新增功能，向后兼容（Minor）
- `x.0.0`：有重大不兼容变更（Major）

### 修改软件名称

`package.json` 中有两处需要同步修改：

```json
{
  "name": "tianma-md-editor",
  "build": {
    "productName": "天马公文系统"
  }
}
```

- `name`：npm 内部标识，建议英文小写加短横线
- `productName`：安装包文件名和桌面快捷方式显示的名称，可以用中文

**其他需要同步修改的位置：**

| 文件 | 修改位置 | 影响的显示内容 |
|---|---|---|
| `main.cjs` | 第 15 行 `title` 字段 | 窗口启动时左上角标题栏 |
| `main.cjs` | 第 62 行 `setTitle` 字符串 | 打开文件后动态更新的标题栏 |
| `index.html` | `<title>` 标签 | 任务栏悬停提示文字 |
| `src/App.jsx` | `app-brand` 处的文案 | 软件内顶栏左上角显示的名称 |

### 修改应用 ID

```json
"build": {
  "appId": "com.yourname.appname"
}
```

应用 ID 用于 Windows 系统唯一标识此程序，影响更新、注册表、快捷方式等。格式为域名反写，改名时建议一并修改。

## 伍 | 如何用 Git 上传到远程平台

以下以 GitHub 为例，Gitee / GitLab 操作完全相同，只需替换仓库地址。

### 确认 `.gitignore` 已配置

推送前，确保项目根目录有 `.gitignore` 文件，避免把大体积目录上传到云端：

```gitignore
node_modules/
dist/
dist_electron/
*.log
.DS_Store
```

### 首次上传（全新仓库）

```powershell
# 1. 初始化 Git 仓库
git init

# 2. 将所有文件加入暂存
git add .

# 3. 提交第一个版本
git commit -m "feat: 初始提交 v1.0.0"

# 4. 关联远程仓库（在 GitHub 新建仓库后获取地址，建库时不要勾选初始化 README）
git remote add origin https://github.com/你的用户名/仓库名.git

# 5. 推送
git push -u origin main
```

> 如果本地分支名是 `master`，把最后一行改为 `git push -u origin master`。

### 日常更新推送（三步走）

```powershell
# 第一步：收集改动
git add .

# 第二步：写提交说明
git commit -m "feat: 新增导出时保留原编号选项"

# 第三步：推送到云端
git push
```

如果提示远端有冲突，先拉取再推送：

```powershell
git pull
git push
```

### 提交信息规范

| 前缀 | 用途 |
|---|---|
| `feat:` | 新增功能 |
| `fix:` | 修复 Bug |
| `style:` | 样式/排版调整，不影响逻辑 |
| `refactor:` | 代码重构 |
| `docs:` | 文档更新 |
| `chore:` | 依赖升级、配置修改等杂项 |

### 打版本标签（发布 Release 时用）

```powershell
# 打标签
git tag v1.1.0

# 推送标签到远端
git push origin v1.1.0
```

在 GitHub 的 Releases 页面可以基于标签发布版本，并附上打包好的 `.exe` 安装包供用户下载。
